Developer Report: Dialog UI Refactor & Preset Integration

Date: 2026-06-16

Status: Completed & Integrated

Executive Summary: The Dialog UI subsystem has been completely refactored to utilize a JSON-driven data model that maps 1-to-1 with the PaperMC Dialog API. This new architecture was seamlessly integrated into the recently developed Preset Configurations system, establishing a "Single Source of Truth" for both preset and inline prompts via the PromptFactory.

1. Motivation & The Problem

Previously, CommandPrompter's dialogs were driven entirely by inline tags (e.g., <d:tab[5]:Player>). While concise, this approach severely limited expressiveness. The Paper Dialog API supports rich features like custom body text, item icons, multi-action button grids, and custom exit buttons. Attempting to cram all these layout configurations into a single inline string would have resulted in an unreadable, overly complex syntax.

Furthermore, the original implementation of <d:tab:T> used T as the input label rather than the Dialog Title, which was counter-intuitive for a full-screen UI picker.

2. Integration with Preset Configurations

To solve the expressiveness problem, the Dialog schema was integrated directly into the new Preset Configurations system (presets.json).

The Preset system uses a discriminator pattern ("type") to parse different prompt definitions (chat, anvil, sign, etc.). We introduced the "type": "dialog" discriminator, allowing server admins to define highly expressive dialogs in JSON and trigger them in-game using the preset syntax: <@preset_id>.

The 1-to-1 Paper API Mapping

The JSON schema was designed to perfectly mirror Paper's builder pattern:

  • base: Maps to DialogBase.builder(). It accepts an array of body elements (plain messages, items) and inputs (text, number, choice).
  • dialog_type: Maps to DialogType. It supports confirmation (with custom confirm/cancel buttons) and multi_action (with columns, static actions, dynamic tab completions, and exit buttons).

3. The "Single Source of Truth" Pipeline

A major architectural goal was to ensure that Inline Prompts and Preset Prompts do not use separate UI rendering logic. To achieve this, we implemented the Transient JSON Translation Strategy.

How it works:

  1. The CommandLineParser extracts an inline tag like <d:tab:Select Player>.
  2. Instead of passing this raw tag to the UI, the InlineTagMapper intercepts it.
  3. The mapper generates a transient JSON DialogPrompt object in memory.
    • For d:tab:T, it sets dialog_type.type = "multi_action", actions_source = "tab_completion", and maps T directly to the title field.
    • For standard inputs (d:text, d:num), it places them in base.inputs and sets the type to confirmation.
  4. This transient JSON object is passed to the PromptFactory, exactly as if it had been loaded from presets.json.

This guarantees that the DialogPromptScreen only ever has to understand one data model: the JSON DialogPrompt record.

4. UI Implementation & Fallback Logic

The DialogPromptScreen.java class was heavily refactored to consume the new JSON models.

Dynamic Tab Completions

When the JSON model specifies "actions_source": "tab_completion", the UI dynamically fetches completions via the TabCompletionService. These completions are mapped into ActionButton objects and appended to the grid after any static actions defined in the JSON. This allows admins to mix static buttons (e.g., "Survival", "Creative") with dynamic player names in the same grid.

Graceful Degradation (The Fallback)

If a multi_action dialog relies on tab completions, but the server returns 0 completions, or the number of completions exceeds the configured MaxButtons threshold, the UI gracefully degrades:

  • It abandons the multi_action layout.
  • It switches to a confirmation layout.
  • It injects a localized warning message into the body (e.g., "Too many options, please type manually").
  • It injects a standard text input field so the user isn't soft-locked.

5. Strict Validation

To prevent configuration errors, strict validation was added to the PromptDefinitionDeserializer. If a user defines a multi_action dialog, the deserializer enforces a strict oneOf rule: the config MUST contain either an actions array (static buttons) OR an actions_source string (dynamic buttons). Providing both without one, or providing neither, throws an immediate IllegalArgumentException during plugin startup, failing fast and protecting the UI from invalid states.

6. Conclusion

The Dialog UI refactor successfully bridges the gap between simple inline commands and highly expressive UI design. By leveraging the Preset Configurations system and the Transient JSON translation strategy, CommandPrompter now fully supports the PaperMC Dialog API without sacrificing the ease of use of its legacy inline syntax.
