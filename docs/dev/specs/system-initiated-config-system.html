Spec: CommandPrompter Server-Initiated Web Preset Editor

Author: Ethan Bacurio (Cyrien)
Version: 1.1.0
Target Plugin: CommandPrompterPaper v3.0.0 (dev.cyr1en.promptpaper)
Target Environment: Dockerized Homelab, Nginx Proxy Manager, Central Redis instance

1. Executive Summary & Context

This document specifies a web-based editor for CommandPrompter's presets.json — the JSON configuration that drives the Prompt Factory (see preset-configuration-spec.html). Server administrators author and edit reusable prompt definitions and post-commands in a browser, with a live in-game preview that renders exactly how each prompt will appear to a player, then push the result back to the game server without a restart.

The editor targets a distributed fleet of Paper servers ($S_0 \dots S_n$). To avoid hosting HTTP/WebSocket servers on individual game instances and bypassing complex inbound port-forwarding, the system uses a Server-Initiated Session model: a central Redis container is the sole middleware for state management and bidirectional communication between the game servers and a centralized Web Application. Redis is an opt-in soft dependency; when no Redis connection is configured, /commandprompter editor reports that the web editor is disabled and the plugin otherwise behaves exactly as today.

2. System Architecture

  • CommandPrompter Paper Plugin (Java): Acts as a Redis client. On demand it serializes the in-memory PresetRegistry contents via PresetGson, pushes the JSON into a session key, and subscribes to a dedicated Pub/Sub channel to receive edited presets. Runs on Paper 1.21.4+ (API 26.1.2) and is Folia-safe — all main-thread mutations are scheduled via the region/global scheduler.
  • Central Redis Server: Stores temporary session states (Key-Value) and routes live updates (Pub/Sub). Entirely internal to the Docker network.
  • Web Backend (Node.js/Python): A stateless API that bridges the SPA frontend with Redis. Exposes endpoints to retrieve a session and push mutations.
  • Web Frontend (Vue.js/React SPA): Served behind Nginx. Parses the presets.json payload, renders a schema-driven form editor for the prompts and post_commands arrays, and renders a live in-game preview alongside each prompt form (see §5).

3. CommandPrompter Integration

The web editor is exposed as a new subcommand on the existing Brigadier /commandprompter tree (alias /cmdp), consistent with the commands documented in docs/user/03-commands.md.

It is the Pub/Sub routing key — the web backend reads it from the session payload to decide which cmdp:notify:<server_id> channel to publish to. Resolution (option C): if unset, the plugin generates a stable UUID once, persists it to plugins/CommandPrompterPaper/.server-id, and reuses it across restarts (warning on boot that the readable name is missing). This guarantees uniqueness and stability even when the admin does nothing, while still allowing a human-readable name when set. Two servers must never share a Server-Id — doing so makes both subscribe to the same channel and silently overwrite each other's presets.

Module boundary: The Redis client, session serializer, and Pub/Sub listener belong in prompt-paper (Paper-specific, game-thread aware). prompt-core must remain agnostic of Redis and of Bukkit/Paper components, preserving the existing module split described in preset-configuration-spec.html.

4. Redis Data Structures & Namespaces

Redis is used for Session Storage and Event Routing. All keys and channels share a cmdp: prefix to avoid collisions with other plugins or services on the shared Redis instance. Per-server isolation is enforced by embedding the Server-Id in the Pub/Sub channel name.

Redis Key / Channel Type Purpose & TTL
cmdp:session:<token> String (JSON) Stores the active presets.json snapshot, its associated Server-Id, and the initiated_by admin UUID. TTL: 300s (5 minutes, idle) — refreshed on each save; deleted on successful apply (single-use).
cmdp:notify:<server_id> Pub/Sub Channel The dedicated channel the specific Paper server subscribes to on boot for live preset updates.

Identifier Roles: Token vs Server-Id

The system uses two distinct identifiers with separate jobs. Confusing them breaks either security or routing.

IdentifierIdentifiesLifetimeWhere it livesJob
Token The edit session (one edit transaction) Per-session; expires with the 300s idle TTL; deleted on successful apply (single-use) Redis key cmdp:session:<token> and the ?session= URL param Opaque lookup handle. The web backend uses it to GET the session payload. Carries no routing info — it must be cryptographically random to resist brute-forcing, so routing can't be encoded into it.
Server-Id The destination Minecraft server Stable, per-server; persists across restarts Inside the session payload (server_id field) and the Pub/Sub channel name cmdp:notify:<server_id> Pub/Sub routing key. The stateless web backend reads it from the payload it just fetched with the token, then publishes to cmdp:notify:<server_id>. Only the server subscribed to that channel receives the update.

The routing chain is: token (in URL) → GET session → read server_id from payload → PUBLISH cmdp:notify:<server_id>. The token fetches; the server-id routes. The stateless backend design depends on having both — the token alone can't tell the backend where to publish.

Why not one global channel? A single cmdp:notify channel that every server subscribes to (filtering by a field in the message) would break isolation at the transport layer: every server would receive and parse every edit from every other server. Per-server channels make isolation architectural — server A cannot receive server B's edit because A never subscribed to B's channel.

5. Live In-Game Preview

The defining feature of the editor is a live preview rendered next to each prompt form. As the admin edits fields, the preview updates in real time to show exactly what the player will see in-game. The preview is a faithful HTML/CSS mock of the Minecraft client UI — it is not a live connection to the server. It reuses the same field values the form is editing, so what you see is what the Prompt Factory will produce.

Prompt type Preview rendering
chat A Minecraft chat-window mock. Shows prompt_text as an incoming chat message. Renders the clickable cancel control (cancel.message, e.g. [&c&l✘]) with its hover_message tooltip, and a simulated player input line. Reflects cancel.clickable and cancel.send.
anvil An Anvil GUI mock: a 3-slot inventory row (left input, right input, result). The left slot renders the left_button.button_icon material with button_text as the item name; the right slot renders right_button the same way. The title bar shows title; the rename text field shows prompt_text. Honors each button's show flag and custom_model_data (rendered as a tooltip tag).
sign A Sign GUI mock: a sign texture with 4 lines. default_lines are pre-filled on the sign; prompt_text is shown as the instruction message above the sign. Editing a default_lines entry re-renders the corresponding sign line immediately.
dialog A Paper Dialog window mock. Renders the title in the title bar, base.body entries (plain messages as text, item bodies as material icons), and one input row per base.inputs entry: text as a text field (with multiline hint), number as a slider bounded by constraints, choice as a dropdown of constraints. The action area reflects dialog_type: confirmation shows confirm_action/cancel_action buttons; multi_action shows the action button grid (or a tab-completion placeholder when actions_source is tab_completion).
player_ui A chest-GUI mock sized to the configured PlayerUI.Size. Renders a paginated grid of player-head items using Skull-Name-Format, with the cancel_button, previous_button, and next_button control items placed at their slot positions (icons from button_icon, labels from button_text). The title shows prompt_text.

5.1 Post-Command Preview

Post-commands have no visual in-game UI, so their preview is a resolved-command card: for each post_command, the editor shows the command string with session-scope placeholders ({input:1}, {input:2}, {player}) highlighted as chips, plus badges for execution_policy (on_complete / on_cancel), execute_as (console / player), and delay_ticks. This mirrors the placeholder semantics in §6.2 of preset-configuration-spec.html.

Severity: A post_command with execute_as: console runs as the server console with {input} substitution. This is the highest-impact field in the editor — a compromised session can inject arbitrary console commands (e.g. op <attacker>, stop). The preview card renders execute_as: console commands with a red warning badge so the admin sees the blast radius at a glance. This severity is the reason the in-game confirmation gate (§7.2, mitigation 1) is mandatory.

5.2 Material & Color Rendering

  • Materials: button_icon / material strings map to Minecraft item textures via a bundled sprite atlas (or a headless item-render service). Invalid materials fall back to PAPER and show a warning badge, matching the plugin's factory fallback (§6.3 of the preset spec).
  • Color & formatting: prompt_text, button labels, and body content may contain legacy ampersand codes (&c) and MiniMessage tags (<red>...</red>). The preview parses both and renders them as styled HTML so the admin sees true colors and gradients.

6. Sequence of Operations

Phase 1: Session Initiation (In-Game)

The administrator logs into a specific backend server (e.g., $S_1$ with Server-Id: survival-01) and initiates the flow.

  1. Admin types /commandprompter editor (or /cmdp editor) in chat. The command requires promptpaper.editor.
  2. If Redis is disabled in config.yml, the plugin short-circuits with "Web editor is disabled (Redis not configured)." and stops.
  3. The plugin generates a cryptographically secure, 16-character alphanumeric token (e.g., xY7v9Pq2Lz5aB1cW) and binds the initiating admin's Minecraft UUID into the session payload as initiated_by (used for audit and in-game confirmation routing).
  4. The plugin serializes the current in-memory PresetRegistry contents into a PresetConfig document using PresetGson.presetGson(). This is the single source of truth — disk is not re-read, so unsaved in-memory state is what the admin edits.
  5. The plugin executes a Redis command to store the session:
    SETEX cmdp:session:xY7v9Pq2Lz5aB1cW 300 '{"server_id": "survival-01", "initiated_by": "<uuid>", "presets": {...}}'
  6. The plugin outputs a clickable URL to the admin as a private message (never broadcast to other players):
    https://config.cyr1en.dev/editor?session=xY7v9Pq2Lz5aB1cW

Phase 2: Frontend Loading & State Hydration

The administrator clicks the link and opens the web application.

  1. The SPA reads the ?session= query parameter from the URL.
  2. Frontend makes a REST/HTTP GET request to the Web Backend: GET /api/session/xY7v9Pq2Lz5aB1cW.
  3. Web Backend queries Redis: GET cmdp:session:xY7v9Pq2Lz5aB1cW.
  4. If the key exists, the backend returns the JSON to the frontend. If it is expired/missing, it returns a 404 (Session Expired).
  5. The frontend validates the presets object against the official JSON schema, then renders a two-pane editor: a schema-driven form on the left (one section per prompt in prompts, one per entry in post_commands, with add/remove/reorder controls), and the live in-game preview on the right (per §5).

Phase 3: Live Editing & Staging a Pending Apply

The administrator modifies the presets in the browser and stages the changes. The web backend does not force the edit onto disk — it publishes a pending apply that the admin must confirm in-game (Phase 4). This is the core anti-hijack control: an attacker holding only the token can stage a payload in Redis but cannot force it onto disk without the admin's authenticated Minecraft session.

  1. As the admin edits any field, the live preview re-renders immediately (debounced). The frontend validates continuously against the schema and disables "Save & Apply" while errors are present.
  2. Admin clicks "Save & Apply". Frontend makes an HTTP POST/PUT request to the Web Backend with the mutated document:
    POST /api/session/xY7v9Pq2Lz5aB1cW/apply
  3. Web Backend re-validates the document against the JSON schema. On failure it returns 400 with field-level errors and does not touch Redis.
  4. Web Backend fetches the session from Redis to verify validity and extract the target server_id and initiated_by.
  5. Web Backend updates the Redis session key with the new state (refreshing the TTL so an active editing session does not expire mid-edit):
    SETEX cmdp:session:xY7v9Pq2Lz5aB1cW 300 '<new_json>'
  6. Web Backend publishes a pending apply to the game server via Pub/Sub. The message includes the initiated_by UUID so the plugin knows which player to prompt for confirmation:
    PUBLISH cmdp:notify:survival-01 '{"pending": true, "initiated_by": "<uuid>", "presets": {...}}'
  7. The frontend transitions to an "Awaiting in-game confirmation" state and may poll GET /api/session/xY7v9Pq2Lz5aB1cW/status for the outcome.

Phase 4: In-Game Confirmation & Hot-Reload

The Paper server gates the disk write behind an in-game confirmation. This is the second factor — the already-authenticated Minecraft session — that closes the hijack-to-RCE gap. The confirm is delivered on the server being edited; the audience is the initiated_by player if present, falling back to any online player with promptpaper.editor on that server (see §7.4 for the full edge-case matrix).

  1. Because survival-01 subscribed to cmdp:notify:survival-01 on boot, its Redis listener thread instantly receives the pending-apply message.
  2. The plugin parses the JSON on an asynchronous thread and validates it with PresetGson.presetGson() into a PresetConfig. If parsing fails, the plugin logs a severe warning, leaves the existing PresetRegistry cache intact, and notifies the admin — the server is never left without presets.
  3. The plugin stores the pending apply in memory with a pending TTL of 300s (separate from the session TTL) and resolves the confirm audience: first the initiated_by player if online on survival-01; otherwise any online player on survival-01 holding promptpaper.editor. If no qualifying player is online, the pending apply is held (not dropped) and re-delivered on the next qualifying PlayerJoinEvent, until the pending TTL expires.
  4. The plugin shows a confirmation prompt to the resolved audience. The prompt is a hardcoded chat/dialog primitive, not a CommandPrompter preset — so it renders even if presets.json is corrupted, which is exactly when a careful confirm matters most: "Web editor wants to apply N changed presets — [Confirm] [Cancel]".
  5. If nobody confirms within the pending TTL, the pending apply is dropped. No disk write occurs. The session-status endpoint reports "confirmation timed out."
  6. On confirm, the plugin re-checks promptpaper.editor at click time (in case it was revoked since session creation), records the confirmer's UUID as confirmed_by in the audit log (separate from initiated_by), then writes the JSON to plugins/CommandPrompterPaper/presets.json on disk and calls PresetRegistry.reload(). reload() re-reads the file, rebuilds the immutable prompt and post-command maps keyed by id, and atomically swaps them in via the volatile reference.
  7. The plugin deletes the Redis session key (single-use apply — the token is now dead, preventing replay): DEL cmdp:session:xY7v9Pq2Lz5aB1cW.
  8. The disk write and cache swap are dispatched on the appropriate Folia scheduler (region/global); the Redis listener never mutates shared plugin state directly from its IO thread.
  9. The plugin sends a success message to the confirmer: "Presets updated via Web Editor." The session-status endpoint reflects success so the browser can update.

7. Security & Edge Cases

7.1 Threat Model

The web backend is stateless, has no user accounts, and is publicly exposed via Nginx. The endpoint POST /api/session/<token>/apply is reachable by anyone on the internet; the only gate is knowledge of a valid session token. The security model is therefore "knowledge of the token = edit access," and the real threats are token leakage, not brute-forcing.

ThreatStatusDetail
Token brute-forcing Solved 6216 ≈ 4.7×1028 possibilities. Infeasible to guess. No real attacker attempts this.
Token in transit Solved HTTPS encrypts the token between browser and web backend.
Token leaked via URL Mitigated (§7.2) URLs leak into browser history, web-backend access logs, and Referer headers on external asset loads (e.g. CDN scripts). HTTPS does not protect the token at these endpoints.
Token leaked via in-game chat Mitigated (§7.2) If the editor link is broadcast (not sent privately), any online player can click it. If the server logs chat, the token is on disk in the chat log.
No binding to the admin Mitigated (§7.2) The token is not tied to the Minecraft player who ran /commandprompter editor. Without a second factor, the backend cannot distinguish the legitimate admin from an attacker holding a leaked token.
Replay of a leaked token Mitigated (§7.2) A leaked token was valid for 3600s — ample time to stage malicious edits. Single-use apply closes this.
Blast radius: RCE via post-commands Mitigated (§7.2) Presets include post_commands with execute_as: console and {input} substitution. A compromised editor is not just "changed prompt text" — it is arbitrary console command execution on the game server. This severity is what makes the thin auth model a real problem, not a theoretical one.

7.2 Mitigations

The baseline security stack. Mitigations 1 and 3 are baked into the apply flow (Phase 3/4); 2, 4, and 5 are deployment and implementation controls.

#MitigationHowWhere
1 In-game confirmation (second factor) On "Save & Apply," the web backend publishes a pending apply, not a direct one. The plugin shows an in-game confirm prompt ("Apply N changed presets — [Confirm] [Cancel]") on the server being edited. Audience: the initiated_by player if present, else any online promptpaper.editor holder on that server (§7.4). Only the in-game click writes disk + reloads. Uses the already-authenticated Minecraft session as the second channel — an attacker with only the token can stage a payload in Redis but cannot force it onto disk without a qualifying game session. confirmed_by is logged separately from initiated_by. Phase 3/4 flow
2 Proxy-level auth (the real auth layer) The token is a session handle, not an auth credential. Put real auth in front of the web backend — Nginx Proxy Manager access lists, basic auth, or Cloudflare Access. The editor URL is then protected by something the attacker doesn't have, independent of the token. Correct separation: token = "which edit session," proxy auth = "are you allowed to edit at all." Deployment (Nginx / Cloudflare)
3 Single-use apply + shorter TTL A successful apply deletes the session key (DEL cmdp:session:<token>), forcing re-initiation for the next edit — a leaked token is dead after one use. TTL reduced from 3600s to 300s (idle); the TTL refreshes on each save, so an active editing session stays alive while an idle leaked token expires in 5 minutes. Phase 1/3/4 flow
4 Leakage-path reduction Send the editor link as a private message to the admin only, never broadcast. Strip ?session= from the web backend's access logs. Set Referrer-Policy: no-referrer on the SPA so the token never leaks via Referer to external assets (CDN scripts, fonts). Plugin (chat send) + Web Backend (log config) + SPA (header)
5 Audit trail Bind initiated_by (admin UUID) into the session payload at creation. Log every apply with timestamp, server_id, and initiated_by. Provides a forensic trail if a compromise occurs. Session payload + plugin log
Severity note — RCE via post-commands: A post_command with execute_as: console runs as the server console with {input} substitution. A bad actor who hijacks a session can inject a preset whose post-command is op <attacker> or stop. Mitigation 1 (in-game confirm) is the control that prevents a staged payload from ever reaching disk, because the admin sees the pending changes and can refuse. This is why the confirm step is mandatory, not optional.

7.3 Edge Cases

  • Automatic Expiration: Redis handles session cleanup natively via the SETEX TTL. Dead sessions leave zero footprint on the system.
  • Server Isolation: Because $S_0$ is only subscribed to cmdp:notify:<S_0's Server-Id>, it is architecturally impossible for a session spawned by $S_1$ to accidentally overwrite $S_0$'s presets.
  • Plugin Reboots: If the game server crashes while the admin is editing, the Pub/Sub message will be lost. To mitigate this, the plugin should SCAN over cmdp:session:* (or maintain an index set) on startup, fetching the latest presets associated with its Server-Id if a live session exists. KEYS is avoided in favor of SCAN to keep Redis non-blocking.
  • Failed Reload Safety: PresetRegistry.reload() throws PresetLoadException on I/O or parse failure and preserves the previous cache. The editor's apply path surfaces this via the session-status endpoint so the admin knows the edit did not take effect.
  • Duplicate IDs: The registry warns on duplicate prompt/post-command ids and keeps the last definition. The frontend warns on duplicate ids during editing, before apply, to prevent silent shadowing.
  • Folia Thread Safety: All disk writes and cache swaps that touch game state are scheduled through the Paper/Folia schedulers; the Redis listener must never mutate shared plugin state directly from its IO thread.
  • Concurrent Edits: If two admins open the same server's session, the last SETEX wins. For multi-admin workflows, each admin should generate their own session.
  • Offline / Absent Initiator: If the initiated_by player is offline or on another backend when the pending apply arrives, the apply is held (not dropped) and re-delivered on the next qualifying PlayerJoinEvent, until the pending TTL expires. See §7.4 for the full matrix.

7.4 Confirmation Gate Edge Cases

Mitigation 1 routes the confirm prompt to a specific audience. The cases below enumerate what goes wrong with that routing and how the design handles each. The guiding rule: never silently drop staged work, and never let a confirm reach disk without a live promptpaper.editor holder on the server being edited.

CaseWhat happensHandling
Initiator disconnects before confirm Admin staged edits, clicked Save & Apply, then alt-tabbed or lost connection. initiated_by player is offline on the target server. Pending apply is held with its 300s pending TTL (not dropped). On the initiator's reconnect (PlayerJoinEvent), if the TTL hasn't expired, the confirm prompt is re-delivered. If any other promptpaper.editor holder is online on the target server, they may confirm in the meantime (delegation).
Initiator disconnects during confirm Admin saw the prompt, then dropped before clicking. Same as above — held and re-delivered on reconnect, or confirmable by another editor. The 300s pending TTL bounds the wait.
Initiator on a different backend (proxied fleet) Fleet is BungeeCord/Velocity-proxied. Admin ran /commandprompter editor on survival-01 but is now playing on creative-02. Bukkit.getPlayer(uuid) on survival-01 returns null — the confirm can't reach them there. Confirm is delivered on the server being edited (where the disk write happens). The initiator must join survival-01 to confirm, or any promptpaper.editor holder on survival-01 may confirm. Cross-server confirm routing (plugin messaging channels) is out of scope (§8).
No qualifying player online Initiator is offline and no other promptpaper.editor holder is on the target server. Pending apply is held until a qualifying player joins or the pending TTL expires. Status endpoint reports "awaiting confirmation, no eligible player online."
Multiple pending applies (same session) Admin clicks Save & Apply, edits more, clicks Save & Apply again before confirming the first. One in-flight pending apply per server. The second /apply supersedes: the plugin cancels the first confirm prompt and shows the latest. The session key's SETEX already reflects the newest presets, so no stale apply can land.
Permission revoked between create and confirm Admin had promptpaper.editor when running /commandprompter editor, but a higher admin revoked it before the confirm click. The plugin re-checks promptpaper.editor at confirm-click time, not just at session creation. If revoked, the confirm is denied with "You no longer have permission to confirm edits." and the pending apply remains for another eligible holder.
Confirm prompt vs. corrupted presets The pending edit itself broke presets.json (or it was already broken). Using a CommandPrompter preset to render the confirm would be circular and could fail. The confirm prompt is a hardcoded chat/dialog primitive, never a preset. It renders independently of PresetRegistry state, so the admin can always review and refuse a bad edit — including one that corrupted the preset system.
Server restart mid-pending The target server crashes/restarts while a pending apply is in flight. Pub/Sub is fire-and-forget, so the message is lost. On reboot, the SCAN recovery (§7.3 "Plugin Reboots") finds the session. If the session carries a pending-apply flag, the plugin re-triggers the confirm flow once a qualifying player is online. The session TTL bounds how long the rebooted apply survives.
Confirm-prompt spam A malicious promptpaper.editor holder spams /commandprompter editor + Save & Apply to flood another admin's chat with confirm prompts. One in-flight pending apply per server at a time — a new apply replaces (and cancels) the previous prompt, so there is never more than one confirm prompt outstanding per server. Session creation can additionally be rate-limited per player.
Initiator in an active prompt session The initiator is mid-way through another CommandPrompter prompt when the confirm arrives. The confirm is a primitive chat click, not a CommandPrompter prompt session, so it does not enter the prompt-session state machine and does not conflict with the active session. The reload (which does cancel active sessions) only happens after a confirm click, by which point the initiator chose to proceed.
Audit fields: initiated_by (who ran /commandprompter editor) and confirmed_by (who clicked confirm) are logged separately. When they differ, the log shows a delegation — useful for forensic review and for the case where the initiator was unavailable and another editor confirmed on their behalf.

8. Out of Scope

  • config.yml / prompt-config.yml editing: This editor is scoped to presets.json only. The plugin's own runtime settings (timeouts, prefixes, UI materials, screen mappings, input-validation aliases) continue to be edited on disk and applied via /commandprompter reload. A future revision may extend the session payload to cover these files.
  • Locale .properties files: Internationalization bundles (see 2026-06-15-i18n-system-design.md) are not edited here; they continue to be managed on disk under plugins/CommandPrompterPaper/locales/.
  • Live Minecraft connection: The preview is a static HTML/CSS mock rendered from the form state. It does not connect to a running server or render real player heads for the player_ui preview (sample heads are used).
  • Permission enforcement per preset: Per preset-configuration-spec.html, presets do not carry per-prompt permission nodes. Permissions are enforced on the root command that triggers the prompt flow, not in the editor.
  • Cross-server confirm routing: The confirm prompt is delivered on the server being edited. Routing a confirm to an initiator who is online on a different backend (via BungeeCord/Velocity plugin messaging or a second Redis round-trip) is out of scope. The initiator must join the target server to confirm, or another editor on the target server confirms.

9.5 Deployment Topology

Plugin: existing Jenkins CI produces the shadow JAR; no change to the release pipeline.

Web: GitHub Actions builds a single Docker image containing both apps — the backend serves the frontend's static assets and the API on one port. One image, one container, behind Nginx Proxy Manager. The backend is a stateless Redis bridge (§2); for a single-admin homelab editor there is no reason to run it as a separate process. If traffic later justifies splitting, the monorepo's apps/ split makes that a deployment change, not a code restructure.

Infrastructure: The web container requires outbound access to the internal Redis container port (default 6379). The web container and Redis container must share a Docker network. Nginx Proxy Manager exposes only the web container's port to the public internet; the Redis container is never exposed publicly. Proxy-level auth (access list, basic auth, or Cloudflare Access) sits in front of the web container (§7.2 mitigation 2).

9.6 CI/CD Boundaries

RepoTriggerOutput
CommandPrompter Push to development; tag release Shadow JAR (Jenkins) + schema asset attached to the GitHub release
CommandPrompter-web Push to main; PR Docker image (GitHub Actions). CI checks the pinned schema version against the latest plugin release and fails on drift.

The only cross-repo coupling is the schema version. A plugin release that changes presets.schema.json should open an automated PR in the web repo to bump the pinned version. If that automation is absent, the web repo's CI drift-check is the fallback — it fails the build before a stale frontend ships.
