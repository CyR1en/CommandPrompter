Dialog Prompt UI Refactor & Schema Migration Spec

Date: 2026-06-16

1. Overview

This document outlines the architectural refactor of the Dialog UI subsystem. It integrates the newly brainstormed expressive Dialog Schema directly into the plugin's unified Prompt Factory architecture defined in docs/dev/specs/preset-configuration-spec.html.

By moving to this schema, both Preset Prompts and Inline Prompts are funnelled through the exact same JSON-driven instantiation pipeline, directly mirroring Paper's native DialogBase and DialogType builders.

2. Architecture Mapping

The JSON structure strictly segregates the configuration into two objects that correspond to Paper's API surface:

  • base: Maps to DialogBase.builder(). Holds the text/item body components and the input widgets.
  • dialog_type: Maps to DialogType. Defines the layout rule (multi-action grid or confirmation prompt).

3. Updated JSON Schema Integration

This overrides and replaces the legacy dialogPrompt definition found in the master preset configuration. The new definition fully supports expressive elements like custom body text, multi-action grids, static buttons, and strict validation. 

"$defs": {
  "actionButton": {
    "type": "object",
    "additionalProperties": false,
    "required": ["label"],
    "properties": {
      "label": { "type": "string", "minLength": 1 },
      "tooltip": { "type": "string" },
      "return": { "type": "string" }
    }
  },
  "dialogTypeConfig": {
    "type": "object",
    "required": ["type"],
    "properties": {
      "type": { "type": "string", "enum": ["multi_action", "confirmation"] }
    },
    "allOf": [
      {
        "if": { "properties": { "type": { "const": "multi_action" } } },
        "then": {
          "required": ["columns"],
          "properties": {
            "columns": { "type": "integer", "minimum": 1 },
            "actions": {
              "type": "array",
              "minItems": 1,
              "items": { "$ref": "#/$defs/actionButton" }
            },
            "actions_source": { "type": "string", "const": "tab_completion" },
            "exit_action": { "$ref": "#/$defs/actionButton" }
          },
          "oneOf": [
            { "required": ["actions"] },
            { "required": ["actions_source"] }
          ]
        }
      },
      {
        "if": { "properties": { "type": { "const": "confirmation" } } },
        "then": {
          "required": ["confirm_action"],
          "properties": {
            "confirm_action": { "$ref": "#/$defs/actionButton" },
            "cancel_action": { "$ref": "#/$defs/actionButton" }
          }
        }
      }
    ]
  },
  "dialogPrompt": {
    "type": "object",
    "additionalProperties": false,
    "required": ["type", "id", "title", "dialog_type"],
    "properties": {
      "type": { "const": "dialog" },
      "id": { "type": "string", "minLength": 1 },
      "title": { "type": "string" },
      "sanitize": { "type": "boolean", "default": true },
      "base": {
        "type": "object",
        "properties": {
          "body": {
            "type": "array",
            "items": {
              "type": "object",
              "required": ["type"],
              "properties": {
                "type": { "type": "string", "enum": ["plain_message", "item"] },
                "content": { "type": "string" },
                "material": { "type": "string" },
                "amount": { "type": "integer", "minimum": 1 }
              }
            }
          },
          "inputs": { 
            "type": "array", 
            "items": { "$ref": "#/$defs/dialogRow" } 
          }
        }
      },
      "dialog_type": { "$ref": "#/$defs/dialogTypeConfig" }
    }
  }
}

4. Inline Syntax Translation Strategy

To preserve the Single Source of Truth model, inline prompts are not instantiated directly. Instead, the CommandLineParser will parse the inline string (e.g., <d:num[1,64]:Amount>) and generate a transient JSON structure in memory. This transient JSON is then handed to the PromptFactory.

Rule 1: Standard Single-Row & Compound Inputs

Tags using text, num, or choice filters map into the base.inputs array. They automatically assume a confirmation dialog type with default button labels sourced from prompt-config.yml.

Inline Syntax Transient JSON Mapping Result
<d:choice[A,B]:Select> 
{
  "type": "dialog",
  "id": "inline-uuid",
  "title": "Command Prompter",
  "base": {
    "inputs": [ { "label": "Select", "input_type": "choice", "constraints": ["A", "B"] } ]
  },
  "dialog_type": {
    "type": "confirmation",
    "confirm_action": { "label": "Confirm" }
  }
}

Rule 2: Tab-Completion (Multi-Action)

If the inline tag uses the d:tab filter, the parser maps it to the multi_action dialog type. Crucially, the T parameter (displayText) is mapped to the Dialog's title.

Inline Syntax Transient JSON Mapping Result
<d:tab:Player> 
{
  "type": "dialog",
  "id": "inline-uuid",
  "title": "Player", // Mapped directly from T
  "dialog_type": {
    "type": "multi_action",
    "columns": 2, // Fetched from prompt-config.yml defaults
    "actions_source": "tab_completion",
    "exit_action": { "label": "Cancel" } // Fetched from config defaults
  }
}

Rule 3: Virtual Title Tags

The legacy <d:title:Custom Title && d:text:Reason> behavior is cleanly preserved. If a d:title sub-tag is detected during parsing, it overrides the root title field in the transient JSON before being stripped from the inputs array.

5. Validation & Fallback Rules

Constraint: A multi_action dialog strictly requires either actions or actions_source. The Gson Deserializer must throw an IllegalArgumentException if both are provided, or if neither are provided.
Tab Fallback: If actions_source: tab_completion returns zero completions, or exceeds the threshold (MaxButtons), the Factory gracefully degrades the UI. It will internally swap the multi_action layout for a confirmation layout, inject a fallback text input into base.inputs, and use the title as the input label.
