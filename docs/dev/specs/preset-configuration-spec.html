CommandPrompter Configuration & Prompt Factory Spec

1. Overview

This specification defines the unified JSON-based configuration and object-generation system for CommandPrompter. It enables administrators to create reusable definitions for prompts and post-commands.

2. System Architecture: The Prompt Factory

This JSON schema serves as the foundational data model for the Prompt Factory. The architecture guarantees a single source of truth for prompt instantiation:

  • Preset Prompts: Pre-defined in the JSON configuration, instantiated by their id.
  • Inline Prompts: Traditional string-based inline prompts (e.g., <prompt_text>) will be internally parsed and translated into this JSON format first.
  • The Factory: Reads the JSON definition and produces the runtime Prompt object. This ensures inline prompts and preset prompts share identical capabilities and underlying logic.

3. Architectural Migration Plan

Transitioning from the legacy PromptTag and ScreenRouter system to the robust PromptFactory pipeline requires specific architectural refactoring.

3.1 Current State Limitations

Currently, prompt construction is strictly Tag-driven and heavily hardcoded:

  • Parsing: The regex parser extracts inline tags (e.g., <a:Why?>) and creates a PromptTag object, which stores minimal flat data (just the key and displayText).
  • Routing: ScreenRouter.java reads the tag's key, looks it up in a basic config map, and decides the ScreenType.
  • Construction: The router calls hardcoded constructors like new AnvilPromptScreen(plugin, player, tag.displayText()). All UI components (buttons, icons, layouts) are rigidly hardcoded inside the screen classes.

3.2 Required Codebase Changes

To implement the JSON Prompt Factory, the architecture must shift from passing a flat PromptTag to passing a rich Data Model (PromptDefinition).

  • Refactor Screen Constructors: The InputScreen implementations (e.g., ChatPromptScreen, AnvilPromptScreen) must be updated to accept the JSON data models (e.g., AnvilPromptDef) rather than plain strings. Internal logic must be rewritten to dynamically map the JSON's cancelBehavior and button_icons instead of using hardcoded values.
  • Introduce the Transient JSON Generator: When the parser detects an inline prompt (<a:Enter Name>), it must generate an ephemeral AnvilPromptDef model (assigning it a transient UUID and injecting default button layouts) and pass it to the factory, instead of a PromptTag.
  • Implement Registry Lookup: When the parser detects a preset tag (<@preset_id>), it queries the PresetRegistry to fetch the predefined PromptDefinition.
  • Transform ScreenRouter into PromptFactory: The existing ScreenRouter logic is to be replaced by the PromptFactory. The factory receives the normalized PromptDefinition, pattern-matches its type, and constructs the appropriate screen using the full JSON definition.
  • Attach Post-Command Pipeline: A new listener must be attached to the prompt flow session to read the PostCommand array, parse placeholders, and dispatch execution based on the completion or cancellation state.

3.3 Legacy Inline Prompt Conversion Pipeline

To achieve a true "Single Source of Truth" via the Prompt Factory, the existing inline prompt syntax (e.g., <a:Enter Name? -ds>) will not be deprecated, but it will no longer execute directly. Instead, it will be mapped into the JSON structure at runtime before instantiation.

  • Type Resolution: The parser reads the prefix key (e.g., a:) and selects the corresponding JSON type discriminator (e.g., anvil). If no key is provided, it defaults to chat.
  • ID Generation: Since the JSON schema requires an id, the parser generates a transient UUID (e.g., inline-5f3a...) for the inline prompt.
  • Data Mapping: The parsed displayText is injected into the prompt_text field. The -ds flag sets sanitize to false.
  • Default Injection: Missing required JSON fields are populated using global default values defined in the plugin's main config.yml.
  • Factory Injection: The completed transient JSON object is passed to the PromptFactory.

4. Tag Resolution Syntax

When executing commands via CommandPrompter, presets are injected using the following syntax:

Type Syntax Example
Prompt Preset <@id> /give <@target_player> <@item_amount>
Post-Command Preset <!@id> /tp spawn <!@notify_admin>

5. Granular Component Specifications

5.1 Prompts

A prompt defines an interactive UI or chat sequence requesting input from the player.

Discriminator Pattern: All prompts are defined as objects in the array and use the type field to determine their implementation. All prompts include a sanitize boolean (default: true) which determines if color codes and formatting symbols should be stripped from the player's input.

Chat Prompt ("type": "chat")

FieldTypeDescription
typeStringMust be exactly chat.
idStringUnique identifier for the factory and <@id> resolution.
prompt_textStringThe text displayed to the player requesting input.
cancelObjectDefines behavior when the player types the cancellation keyword. Includes send, message, clickable, and hover_message.
sanitizeBooleanWhether to strip color codes from input.

Anvil Prompt ("type": "anvil")

FieldTypeDescription
typeStringMust be exactly anvil.
idStringUnique identifier.
titleStringInventory title of the Anvil GUI.
prompt_textStringDefault text placed in the Anvil text field.
left_button / right_buttonObjectItems placed in the anvil slots. Requires show, button_text, button_icon, button_hover_text, and custom_model_data.
sanitizeBooleanWhether to strip color codes from input.

Player UI Prompt ("type": "player_ui")

FieldTypeDescription
typeStringMust be exactly player_ui.
idStringUnique identifier.
prompt_textStringInventory title requesting player selection.
filterStringOptional logic to filter displayed players.
cancel_button / previous_button / next_buttonObjectPagination and control buttons. Requires slot, show, button_icon, etc.
sanitizeBooleanWhether to strip color codes from input.

Sign Prompt ("type": "sign")

FieldTypeDescription
typeStringMust be exactly sign.
idStringUnique identifier.
prompt_textStringMessage sent to the player explaining what to write.
default_linesArray[String]Up to 4 lines of text pre-filled on the sign GUI.
sanitizeBooleanWhether to strip color codes from input.

Dialog Prompt ("type": "dialog")

FieldTypeDescription
typeStringMust be exactly dialog.
idStringUnique identifier.
titleStringInventory title of the Dialog GUI.
baseObjectDefines the body (plain messages, items) and inputs (text, number, choice).
dialog_typeObjectDefines the layout: multi_action (with columns, actions, or tab_completion) or confirmation.
sanitizeBooleanWhether to strip color codes from input globally.

5.2 Post-Commands

Post-commands define side-effects triggered after the lifecycle of a prompt flow session completes or is cancelled.

FieldTypeDescription
idStringUnique identifier for <!@id> resolution.
commandStringThe raw command to execute (without the leading slash).
execution_policyEnumWhen to trigger: on_complete or on_cancel.
execute_asEnumContext of execution: console or player.
delay_ticksInteger(Optional) Delay in server ticks before execution. Defaults to 0.

6. Implementation Rules & Edge Cases

To ensure a robust implementation, the Prompt Factory and parser must adhere to the following behavioral rules:

6.1 Inline Prompt ID Generation

Because the JSON schema explicitly requires an id, inline prompts (which do not have explicit IDs defined by the user) must automatically generate an ephemeral identifier during the parsing phase. Solution: The parser should generate a transient UUID (e.g., inline-<uuid>) and assign it to the JSON definition before passing it to the Factory. This ensures the Factory logic remains strictly uniform without requiring schema exemptions.

6.2 Post-Command Placeholders (Session Scope)

Post-commands are rendered useless if they cannot reference the data captured during the prompt flow. Furthermore, post-commands operate at the Session Level, meaning a single command string may have triggered multiple distinct prompts. Solution: The command string must parse the following indexed variables immediately prior to execution:

  • {input} or {input:1}: The string value resolved from the first prompt in the command sequence.
  • {input:2}, {input:3}, etc.: The string values resolved from subsequent prompts in the command sequence.
  • {player}: The name of the player who triggered the prompt flow.
  • PlaceholderAPI (PAPI) Support: Standard %...% PAPI placeholders should be parsed relative to the executing player.

6.3 Material Icons Parsing

The button_icon field in UI components is defined as a String. Solution: This string must strictly map to valid Bukkit Material enum names (case-insensitive, e.g., DIAMOND_SWORD or minecraft:stone). If an invalid material is provided, the factory should default to a fallback material (e.g., PAPER) and log a non-fatal warning to the server console rather than crashing the prompt.

6.4 Missing or Invalid Tag Resolution

If a player triggers a command that contains an unresolved tag (e.g., /give <@non_existent_id>). Solution: The system must Fail Fast. The command flow must be immediately cancelled. The plugin should send a localized error message to the user (e.g., "Configuration error: A required prompt is missing.") and log a severe warning to the console detailing the missing ID. Under no circumstances should the literal tag be passed to the underlying command.

6.5 Caching and Hot-Reloading

Disk I/O should not occur during command execution. Solution: The presets.json file must be read, validated against the schema, and fully cached into memory during the plugin's onEnable() phase. To apply JSON updates without restarting the server, the plugin must support a /commandprompter reload command that flushes the factory cache and rebuilds it from disk.

7. Implementation Scopes

The development of this feature is divided into 5 clear scopes to allow structured and parallel implementation.

Scope 1: Data Models & Deserialization

  • Create Java Record/POJO classes for ChatPrompt, AnvilPrompt, PlayerUIPrompt, SignPrompt, DialogPrompt, and PostCommand.
  • Implement JSON deserialization logic using the type field discriminator pattern to map to the correct implementation.
  • Implement Bukkit Material mapping logic and fallback defaults.

Scope 2: Configuration Management & Caching

  • Create the PresetRegistry to hold loaded configurations in memory.
  • Implement the plugin startup logic (reading and parsing presets.json).
  • Implement the /commandprompter reload behavior for safe, runtime cache rebuilding.

Scope 3: The Prompt Factory

  • Build the PromptFactory class that receives a JSON data model and instantiates a functional runtime Prompt object.
  • Implement ephemeral UUID assignment (inline-<uuid>) for translating legacy inline prompts into the factory pipeline.

Scope 4: Command Tag Resolution & Interception

  • Update regex parsers to detect and isolate <@id> and <!@id>.
  • Implement the fail-fast flow cancellation and error-logging when an ID does not exist in the PresetRegistry.

Scope 5: Post-Command Execution Engine

  • Build the queue attached to prompt flow sessions to track pending post-commands.
  • Implement pre-execution placeholder parsing ({input:1}, {player}, and PAPI).
  • Implement the dispatcher supporting execute_as context switching and Bukkit/Folia-safe delay_ticks scheduling.

8. Out of Scope & Module Boundaries

The following items are explicitly excluded from this specification and implementation phase:

  • Module Strictness: To maintain clean domain boundaries, pure JSON definition mapping, UI-concept Records (Anvil, Sign), and Bukkit Material fallback logic MUST reside in the prompt-paper module. The prompt-core module must remain completely agnostic to game-specific components and dependencies.
  • Input Validation logic: Native input validators (Regex checking, online player validation) have been removed from the schema and are not supported natively by the factory.
  • Database Integration: Configurations will strictly reside in flat file formats (JSON). No database (SQL/Redis) sync features will be built for preset storage.
  • Per-Prompt Permission Nodes: The presets do not check if a player has permission to see or use a specific prompt. Permissions should be enforced on the root command that triggers the prompt flow.
  • GUI Framework Implementation: The factory will rely entirely on the existing CommandPrompter UI abstraction layer (prompt-ui-api). No new inventory API mechanics are to be developed.

9. Official JSON Schema

The following schema defines the strict structural requirements for the presets.json configuration file, utilizing a discriminator pattern ("type") rather than nested object wrappers for clean Java deserialization.

{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "title": "CommandPrompter JSON Configuration",
  "type": "object",
  "additionalProperties": false,
  "required": ["prompts", "post_commands"],
  "properties": {
    "prompts": {
      "type": "array",
      "description": "A list of prompt configurations using the 'type' discriminator.",
      "items": {
        "type": "object",
        "oneOf": [
          { "$ref": "#/$defs/chatPrompt" },
          { "$ref": "#/$defs/anvilPrompt" },
          { "$ref": "#/$defs/playerUiPrompt" },
          { "$ref": "#/$defs/signPrompt" },
          { "$ref": "#/$defs/dialogPrompt" }
        ]
      }
    },
    "post_commands": {
      "type": "array",
      "description": "Commands executed after prompt flow events.",
      "items": { "$ref": "#/$defs/postCommand" }
    }
  },
  "$defs": {
    "cancelBehavior": {
      "type": "object",
      "additionalProperties": false,
      "required": ["send", "message", "clickable", "hover_message"],

    },

        "show": { "type": "boolean" },
        "button_text": { "type": "string", "minLength": 1 },
        "button_icon": { "type": "string", "minLength": 1 },
        "button_hover_text": { "type": "string", "minLength": 1 },
        "custom_model_data": { "type": "integer", "minimum": 0 }
      }
    },
    "uiButton": {
      "type": "object",
      "additionalProperties": false,
      "required": ["show", "slot", "button_text", "button_icon", "button_hover_text", "custom_model_data"],
      "properties": {
        "show": { "type": "boolean" },
        "slot": { "type": "integer", "minimum": 0, "maximum": 53 },
        "button_text": { "type": "string", "minLength": 1 },

      "additionalProperties": false,

        "prompt_text": { "type": "string", "minLength": 1 },

          "items": { "type": ["string", "number"] }

      }
    },
    "actionButton": {
      "type": "object",
      "additionalProperties": false,
      "required": ["label"],
      "properties": {
        "label": { "type": "string", "minLength": 1 },
        "tooltip": { "type": "string" },
        "return": { "type": "string" }
      }
    },
    "dialogTypeConfig": {
      "type": "object",
      "required": ["type"],

      "allOf": [
        {
          "if": { "properties": { "type": { "const": "multi_action" } } },
          "then": {
            "required": ["columns"],
            "properties": {
              "columns": { "type": "integer", "minimum": 1 },
              "actions": {
                "type": "array",
                "minItems": 1,
                "items": { "$ref": "#/$defs/actionButton" }

              { "required": ["actions"] },
              { "required": ["actions_source"] }
            ]
          }
        },
        {
          "if": { "properties": { "type": { "const": "confirmation" } } },
          "then": {
            "required": ["confirm_action"],
            "properties": {
              "confirm_action": { "$ref": "#/$defs/actionButton" },
              "cancel_action": { "$ref": "#/$defs/actionButton" }
            }
          }
        }
      ]
    },
    "dialogPrompt": {
      "type": "object",
      "additionalProperties": false,
      "required": ["type", "id", "title", "dialog_type"],
      "properties": {
        "type": { "const": "dialog" },
        "id": { "type": "string", "minLength": 1 },
        "title": { "type": "string", "minLength": 1 },
        "sanitize": { "type": "boolean", "default": true },
        "base": {
          "type": "object",
          "properties": {
            "body": {
              "type": "array",
              "items": {
                "type": "object",
                "required": ["type"],
                "properties": {
                  "type": { "type": "string", "enum": ["plain_message", "item"] },
                  "content": { "type": "string" },
                  "material": { "type": "string" },

              }
            },
            "inputs": { 
              "type": "array", 
              "items": { "$ref": "#/$defs/dialogRow" } 
            }
          }
        },
        "dialog_type": { "$ref": "#/$defs/dialogTypeConfig" }
      }
    }
  }
