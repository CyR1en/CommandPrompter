Spec: CommandPrompter Web Preset Editor v2

Author: Ethan Bacurio (Cyrien)
Version: 2.0.0
Supersedes: system-initiated-config-system.html (v1)
Target Plugin: CommandPrompterPaper v3.0.0 (dev.cyr1en.promptpaper)
Reference Architecture: LuckPerms /lp editor (bytebin + bytesocks), with improvements

1. Executive Summary

This revision specifies a web-based editor for CommandPrompter's presets.json (the Prompt Factory config defined in preset-configuration-spec.html). The admin edits presets in a browser with a live in-game preview, then applies changes to the game server without a restart.

The architecture is outbound-only: the plugin makes an outbound WSS connection to a hardcoded public relay endpoint. No inbound ports on the game server. No Redis credentials or internal service addresses in user config. The relay is a dumb pipe — it routes signed, encrypted frames between plugin and browser. It stores nothing. Security is end-to-end between plugin and browser.

Key improvements over the LuckPerms reference and v1:

  • End-to-end encryption — the relay never sees plaintext presets (which may contain execute_as: console post-commands).
  • Risk-tiered apply — routine edits auto-apply; console-command edits require in-game confirmation with a diff preview.
  • Ed25519 signatures — deterministic nonce, no ECDSA nonce-reuse risk.
  • 256-bit token — single identifier for the WSS channel and initial encryption key.
  • Session persistence — survives plugin restart; admin doesn't lose work.
  • Single unified relay — one service, one endpoint, one failure mode (vs. LuckPerms' bytebin + bytesocks).
  • Local fallback — works with zero network connectivity.
  • Plugin-side schema validation — defense in depth against malformed payloads.
  • Structured audit trail — queryable JSONL log of every apply.

The only user-facing config is a single boolean in config.yml:

No URLs, no Redis, no server addresses. When false, the editor subcommand is a no-op.

3.3 Hardcoded Endpoints

Endpoint URLs live in a classpath resource, not in user config:

For self-hosters: an optional override file at plugins/CommandPrompterPaper/editor-endpoints.override.properties takes precedence over the classpath resource. This is a dev/self-host mechanism, not documented in the user config reference.

3.4 Module Boundary

All new code lands in prompt-paper. prompt-core stays agnostic of HTTP, WebSocket, crypto sessions, and Bukkit/Paper components. The relay client, crypto, session store, and editor command belong in prompt-paper.

4. Cryptography & Authentication

The security model has three layers: end-to-end encryption (confidentiality), Ed25519 signatures (authenticity), and browser-fingerprint trust (first-connection authorization). The relay is untrusted — it sees only ciphertext and opaque signed frames.

4.1 Session Token

The plugin generates a 256-bit token: base64url(SecureRandom.getBytes(32)) — 43 characters, ~4.3×1077 keyspace. This single value serves two roles:

RoleHow
WSS channel idThe relay routes frames on channel /ws/{token}. Plugin and browser join the same channel.
Initial encryption key materialK_initial = HKDF-SHA256(ikm=token, salt="cmdp-e2e-v1", info="initial-transport") → 32-byte AES-256-GCM key. Used to encrypt the initial preset push before the key exchange completes.

The token is delivered out-of-band via a private in-game chat message. The relay sees the channel id (which is the token) but cannot derive the AES key from it without the HKDF salt/info constants (which are in the plugin and browser code, not on the relay).

4.2 End-to-End Encryption

Two phases: initial (token-derived key) and session (X25519 ECDH-derived key for forward secrecy).

Phase 1: Initial Transport (before handshake)

  1. Plugin derives K_initial from the token via HKDF.
  2. Plugin encrypts the serialized PresetConfig: AES-256-GCM(K_initial, plaintext, nonce=random-12-bytes, aad=token).
  3. Plugin opens WSS to {relay.url}/ws/{token} and waits for the browser to join.
  4. When the browser connects and the trust handshake completes, the plugin pushes the encrypted presets as a single WSS frame.
  5. Browser derives the same K_initial from the token in the URL and decrypts.

Phase 2: Session Transport (after handshake, forward secrecy)

sequenceDiagram
    participant P as Plugin
    participant B as Browser
    participant R as Relay
    P->>R: WSS connect to channel
    B->>R: WSS connect to channel
    B->>P: HELLO with X25519 pubkey and Ed25519 pubkey

    Note over P,B: K session equals HKDF of shared secret
    Note over P,B: all future frames use K session
  1. Plugin generates an X25519 keypair: (priv_x25519_p, pub_x25519_p).
  2. Browser generates an X25519 keypair: (priv_x25519_b, pub_x25519_b).
  3. Both exchange X25519 public keys via the signed HELLO frames (see §4.3).
  4. Shared secret: SS = X25519(priv_x25519_p, pub_x25519_b) = X25519(priv_x25519_b, pub_x25519_p).
  5. Session key: K_session = HKDF-SHA256(ikm=SS, salt=token, info="session-transport") → 32 bytes.
  6. All subsequent frames (edited presets, follow-up sessions) are encrypted with K_session. The initial K_initial is retired.

Nonce Management

AES-256-GCM requires that each (key, nonce) pair is used exactly once — nonce reuse with the same key is catastrophic. The scheme is counter-based with role-prefixed counters to guarantee no collision between plugin and browser:

PhaseKeyNonce schemeCounter range
Initial (K_initial)Token-derivedRandom 12-byte nonce per frame. Only one frame is sent under K_initial (the initial presets push), so collision probability is zero by construction.N/A (single use)
Session (K_session)ECDH-derived64-bit counter, big-endian, zero-padded to 12 bytes. Plugin uses even counters (0, 2, 4...); browser uses odd counters (1, 3, 5...). Counter increments per encrypted frame sent.Plugin: even. Browser: odd. Wraps at 264 (infeasible for config-edit traffic).

Each side tracks its own send counter in memory for the session. Counters are not sent in the frame — the receiver tracks its own receive counter per sender. If a frame arrives with an unexpected counter (gap or regression), the receiver rejects it with change-response {state: "rejected", reason: "nonce_out_of_order"} and closes the channel — this indicates either a bug or a replay attack.

Forward secrecy: If the token is later compromised (e.g., from a browser history leak), past sessions encrypted with K_session cannot be decrypted — the X25519 ephemeral keys are gone. Only the initial upload (encrypted with K_initial) would be vulnerable, and it contains the original presets, not the edits.

4.3 Ed25519 Frame Signatures

Every WSS frame is signed by the sender's Ed25519 private key and verified by the receiver against the sender's known public key. Ed25519 uses deterministic nonce derivation — no nonce-reuse vulnerability (unlike ECDSA P-256). The msg field contains the AES-256-GCM ciphertext of the inner JSON message; the receiver decrypts before verifying the signature. The one exception is hello, which is MAC'd (not signed) because no peer key is known yet — see §4.4.

// Frame envelope (JSON, sent as WSS text)
{
  "msg": "<AES-256-GCM ciphertext, base64>",
  "sig": "<base64(Ed25519.sign(privKey, msg_ciphertext))>",
  "nonce": "<base64(12-byte AES-GCM nonce)>"
}
KeyGenerated byLifetimeStored
Plugin Ed25519 keypairPlugin, at session creationPer-session (in-memory)Not persisted — regenerated each session
Browser Ed25519 keypairBrowser, on first visitPersistentIndexedDB (or localStorage fallback)

The plugin's Ed25519 public key is embedded in the initial encrypted presets frame (inside the ciphertext, readable only by the browser). The browser's Ed25519 public key is sent in the HELLO frame. After the trust handshake (§4.4), both sides know each other's public key and verify every subsequent frame.

4.4 Browser-Fingerprint Trust

First connection from a browser requires in-game admin approval. Subsequent connections from the same browser are auto-trusted.

  1. Browser sends HELLO frame containing: {nonce, sessionId, browser, publicKey: Ed25519, x25519PublicKey, mac: HMAC-SHA256(K_initial, sessionId || publicKey || x25519PublicKey)}. The MAC is keyed by K_initial, which the relay does not have — this proves the sender knows the token without revealing it, and prevents a malicious relay from substituting its own keys (MITM). The HELLO is the one frame allowed without an Ed25519 signature (no peer key is known yet), but the MAC binds it to the token-holder.
  2. Plugin verifies the MAC against K_initial. If it fails, the plugin rejects the connection and closes the channel — the sender doesn't know the token.
  3. Plugin checks its trust store for SHA-256(browser Ed25519 public key).
  4. If trusted → send HELLO_REPLY {state: "accepted", x25519PublicKey: plugin} (signed with the plugin's Ed25519 key, MAC'd with K_initial), proceed with X25519 key exchange.
  5. If untrusted → send HELLO_REPLY {state: "untrusted"} (signed + MAC'd). Plugin prompts admin in-game: "Untrusted browser (<user-agent>) wants to connect — /commandprompter trusteditor <nonce>".
  6. Admin types /commandprompter trusteditor <nonce>. Plugin stores SHA-256(browser Ed25519 pubkey) in trust store, sends HELLO_REPLY {state: "accepted", ...}.
  7. If rejected (different key from an already-connected browser on this session) → send HELLO_REPLY {state: "rejected"}, close socket.

Trust store: plugins/CommandPrompterPaper/.editor-truststore.json — a JSON file mapping playerUUID → [SHA-256(browser pubkey), ...]. Persisted across restarts. An admin can clear trust with /commandprompter trusteditor clear.

4.6 Frame Protocol

Every WSS text frame uses the envelope structure from §4.3: {msg, sig, nonce}. The msg field is the AES-256-GCM ciphertext of an inner JSON message. After decryption, the inner JSON contains a type field and type-specific payload. The relay sees only the ciphertext envelope — it cannot read the message type or payload.

Frame ordering: WSS guarantees in-order delivery per connection. The relay routes frames in the order received. No sequence numbers are needed — the counter-based nonce scheme (§4.2 Nonce Management) implicitly enforces ordering on encrypted frames.

5. Relay Service

A single unified WSS relay. The relay is a dumb pipe: it routes signed, encrypted frames between plugin and browser on the same channel. It stores nothing — no paste store, no TTL, no persistence layer. It never decrypts, validates, or authenticates. It can be compromised without exposing preset contents or enabling forged applies.

5.1 API Surface

EndpointMethodRequestResponseNotes
/ws/{token} GET (Upgrade) Upgrade: websocket WSS connection on channel {token} Routes text frames between connected clients on the same channel. Max 2 clients per channel (plugin + browser). Third client is rejected with 4003 "channel full".
/ + static GET — SPA static assets The relay also serves the editor SPA's built static files. One container, one port.

5.2 WSS Relay Rules

  • Max clients per channel: 2. The first client is the plugin; the second is the browser. A third connection attempt is rejected with 4003 "channel full".
  • Frame format: text frames only (JSON). Binary frames are rejected.
  • Max frame size: 5 MB. Presets JSON is typically <100 KB, but the full 5 MB ceiling accommodates large preset documents. The relay buffers one frame per channel (2 clients) — memory pressure is trivial.
  • Idle timeout: 60s without any frame → close with 4001 "idle". The plugin sends ping frames every 10s to keep the channel alive.
  • No storage: the relay holds no state between frames. When both clients disconnect, the channel is gone. There is no paste to expire, no key to evict, no persistence layer to operate.

5.3 Rate Limiting & Abuse Prevention

SurfaceLimitEnforcement
WSS connections5 concurrent per source IPReject with 4003
WSS messages per channel100 per minuteClose with 4003
WSS frame size5 MB maxClose with 4009 "message too big"
The relay cannot distinguish the plugin from an attacker (no pre-shared secret). Rate limits prevent flooding. The 256-bit token makes session-guessing infeasible. E2E encryption means a compromised relay gains nothing but ciphertext. Signed frames mean a compromised relay cannot forge applies.

6.1 Post-Command Preview & RCE Severity

Post-commands render as a resolved-command card: the command string with {input:N}/{player} placeholders as chips, plus badges for execution_policy, execute_as, and delay_ticks.

RCE severity: A post_command with execute_as: console runs as the server console with {input} substitution. A compromised session can inject op <attacker> or stop. The preview renders console commands with a red warning badge. This severity is why risk-tiered apply (§8) gates console-command edits behind in-game confirmation.

6.2 Material & Color Rendering

  • Materials: button_icon/material strings map to Minecraft item textures via a bundled sprite atlas. Invalid materials fall back to PAPER with a warning badge (matches the plugin's factory fallback).
  • Color: prompt_text, labels, and body content may contain legacy &-codes and MiniMessage tags. The preview parses both and renders styled HTML.

7. Sequence of Operations

Phase 1: Session Initiation (In-Game)

  1. Admin types /commandprompter editor. Requires promptpaper.editor. If Web-Editor.Enabled is false, short-circuits with "Web editor is disabled."
  2. Plugin generates 256-bit token: base64url(SecureRandom.getBytes(32)).
  3. Plugin generates Ed25519 keypair (signing) and X25519 keypair (ECDH) for this session.
  4. Plugin serializes in-memory PresetRegistry via PresetGson.presetGson() into a PresetConfig document. Binds initiated_by (admin UUID) and plugin Ed25519/X25519 public keys into the payload.
  5. Plugin derives K_initial from the token via HKDF. Encrypts the payload with AES-256-GCM.
  6. Plugin opens WSS to {relay.url}/ws/{token}. Sends a ping every 10s to keep alive. Waits for the browser to join.
  7. Plugin persists the session to .editor-sessions.json (survives restart — see §9.3).
  8. Plugin sends a private message to the admin with the editor URL: {editor.url}?t={token}. Never broadcast.

Phase 2: Browser Loading & Crypto Handshake

  1. Browser opens {editor.url}?t={token}. SPA reads the token from the query param. The SPA derives the WSS URL from its own origin (wss://{window.location.host}/ws/{token}) — no separate relay URL config needed since the SPA is served by the relay itself.
  2. SPA derives K_initial from the token (same HKDF parameters as the plugin).
  3. SPA opens WSS to wss://{window.location.host}/ws/{token}. Sends HELLO frame: {nonce, sessionId: token, browser: navigator.userAgent, publicKey: Ed25519, x25519PublicKey}.
  4. Plugin checks trust store for SHA-256(browser Ed25519 pubkey):
    • Trusted → sends HELLO_REPLY {state: "accepted", x25519PublicKey: plugin}. Both sides derive K_session via X25519 ECDH + HKDF. SPA switches to K_session for all future frames.
    • Untrusted → sends HELLO_REPLY {state: "untrusted"}. Plugin prompts admin in-game with nonce and browser user-agent. Admin types /commandprompter trusteditor <nonce>. Plugin stores the key hash, sends HELLO_REPLY {state: "accepted", ...}. Key exchange proceeds.
  5. Plugin pushes the encrypted presets as a single WSS frame (encrypted with K_initial). SPA decrypts, parses the PresetConfig + embedded plugin public keys, and validates against the JSON schema. On failure, shows a schema-error screen and stops.
  6. SPA renders the two-pane editor: schema-driven form (left) + live preview (right).

Phase 3: Editing & Diff Preview

  1. Admin edits presets. Live preview re-renders on every field change (debounced 200ms).
  2. SPA validates continuously against the schema. "Apply" button is disabled while errors are present.
  3. SPA computes a structured diff against the original document: added/removed/changed presets and post-commands, with field-level detail.
  4. SPA shows the diff in a review panel. For post_commands with execute_as: console, the full command string is displayed with a red RCE badge.
  5. Admin clicks "Apply". SPA encrypts the edited document with K_session and sends a signed change_request frame over WSS: {type: "change_request", presets: PresetConfig}. The envelope encryption handles confidentiality — the presets are plaintext at the message level.

Phase 4: Risk-Tiered Apply

The plugin receives the change-request frame, decrypts it with K_session, validates the presets against the schema, classifies the risk, and applies — with or without in-game confirmation depending on the risk tier. See §8 for the full classification and apply paths.

Phase 5: Plugin Ingestion (Hot-Reload)

  1. Plugin writes the validated JSON to plugins/CommandPrompterPaper/presets.json on disk.
  2. Plugin calls PresetRegistry.reload(). Re-reads the file, rebuilds immutable prompt/post-command maps keyed by id, atomically swaps via volatile reference. On PresetLoadException, previous cache is preserved and the admin is notified.
  3. Disk write and cache swap are dispatched on the Folia scheduler (region/global). The WSS listener never mutates shared state on its IO thread.
  4. Plugin logs the apply to the audit trail (§10.3).
  5. Plugin sends a signed change-response frame: {state: "applied"}. The plugin then pushes the post-apply state as a new encrypted frame so the browser can continue editing without re-initiating.
  6. Plugin sends a success message to the admin: "Presets updated via Web Editor."
  7. Plugin keeps the WSS channel open for continued editing. The session is removed from .editor-sessions.json only when the channel closes (both sides disconnect or idle timeout).

8. Risk-Tiered Apply

Not every edit needs in-game confirmation. Routine edits (prompt text, UI labels) auto-apply for convenience. Edits that can execute console commands require the admin to review and confirm in-game. The classification is computed from the diff at the plugin side — the frontend's risk indicator is advisory only.

8.2 High-Risk Confirmation Flow

  1. Plugin identifies the high-risk changes from the diff and extracts the exact command strings from every added/modified console post_command.
  2. Plugin resolves the confirmation audience: the initiated_by player if online on this server; otherwise any online player with promptpaper.editor. If no qualifying player is online, the pending apply is held (not dropped) and re-delivered on the next qualifying PlayerJoinEvent, until the pending TTL (300s) expires.
  3. Plugin shows a hardcoded chat/dialog primitive (never a CommandPrompter preset — must render even if presets.json is corrupted): "Web editor wants to apply N changes, including M console commands: [Review] [Confirm] [Cancel]".
  4. [Review] paginates the exact command strings in chat. [Confirm] proceeds. [Cancel] drops. 60s timeout drops.
  5. On confirm, plugin re-checks promptpaper.editor at click time (in case it was revoked). Records confirmed_by UUID in the audit log (separate from initiated_by).
  6. Plugin writes disk + reloads. Sends change-response {state: "applied"} to the browser.
One in-flight pending apply per server. A second /apply supersedes the first: the plugin cancels the old confirm prompt and shows the latest. No more than one confirm prompt is ever outstanding per server.

8.3 Diff Preview

The diff is shown in the browser before the admin clicks Apply, and again in-game for high-risk confirms. The browser diff is structured:

{
  "added_prompts": [{"id": "new_prompt", "type": "chat"}],
  "removed_prompts": [{"id": "old_prompt"}],
  "changed_prompts": [{"id": "ban_form", "fields": ["dialog_type.type"]}],
  "added_post_commands": [{"id": "log_reason", "execute_as": "console",
    "command": "discord broadcast {player} punished {input:1}"}],
  "removed_post_commands": [],
  "changed_post_commands": []
}

Console post_commands in the diff are rendered with the full command string and a red RCE badge, both in the browser diff panel and in the in-game review.

9. Fallback Modes

Two modes of operation, selected automatically based on connectivity. Each mode degrades gracefully.

9.1 Live Mode (default)

WSS relay is reachable. Full bidirectional communication. One-click apply with risk-tiered confirmation. This is the primary mode described in §7.

9.2 Local Mode (relay completely unreachable)

If the relay is down or the server has no internet access:

  1. /commandprompter editor detects relay unreachability (WSS connect fails or times out after 5s). Plugin informs the admin: "Relay unreachable. Use local export/import mode."
  2. Admin runs /commandprompter export. Plugin writes the current presets to plugins/CommandPrompterPaper/presets-export.json (plaintext, no encryption — it's a local file).
  3. Admin edits presets-export.json manually or with a local JSON editor.
  4. Admin runs /commandprompter import presets-export.json. Plugin validates against presets.schema.json, classifies risk, and applies (with confirmation if high-risk).

9.3 Session Persistence Across Restarts

Sessions are persisted to plugins/CommandPrompterPaper/.editor-sessions.json on creation and removed on channel close. On plugin startup:

  1. Plugin loads the session map from disk.
  2. For each live session, the plugin reopens the WSS connection to {relay.url}/ws/{token}.
  3. If the browser reconnects (its tab is still open or reopened), the trust store is persisted so the browser's fingerprint is still recognized. The plugin pushes the current presets again and editing resumes.
  4. If the browser doesn't reconnect within the idle timeout (60s), the channel closes and the session is removed. The admin is notified on next join: "Your web editor session expired (server restarted). Run /commandprompter editor to start a new one."

10. Security & Edge Cases

10.1 Threat Model

ThreatStatusDetail
Token brute-forcingSolved256-bit token. 4.3×1077 keyspace. Infeasible.
Token in transitSolvedHTTPS/WSS encrypts the channel. Token is in a URL query param — see leakage mitigations below.
Compromised relay reads presetsSolvedE2E encryption (AES-256-GCM). Relay routes only ciphertext frames — it stores nothing. Even a compromised relay operator cannot read preset contents.
Compromised relay forges an applySolvedEd25519 signatures on every WSS frame. Relay cannot sign frames — it doesn't have either party's private key. Forged frames are rejected.
Token leaked via URL (history, logs, Referer)MitigatedPrivate in-game send only. Relay strips ?t= from access logs. SPA sets Referrer-Policy: no-referrer. Forward secrecy limits damage to the initial preset push only.
Trusted browser compromisedMitigatedRisk-tiered apply (§8). A compromised trusted browser can stage edits but cannot force high-risk (console-command) edits to disk without in-game confirmation.
Replay of a leaked tokenMitigatedThe token enables channel access and decrypts the initial preset push (encrypted with K_initial). However, subsequent frames are encrypted with K_session (ECDH-derived), which the token alone cannot derive — forward secrecy protects post-handshake traffic. The attacker would also need the browser's Ed25519 private key to sign frames. Channel idle timeout (60s) limits the window.
Relay DoS (flood fake sessions)MitigatedRate limiting per IP (§5.3). 256-bit token makes guessing infeasible. E2E encryption means fake sessions contain ciphertext the attacker can't read either.
Malformed payload reaches diskSolvedPlugin-side schema validation against presets.schema.json before writing to disk. Defense in depth — the browser validates too, but the plugin doesn't trust the browser.
RCE via console post-commandsMitigatedHigh-risk tier requires in-game confirmation with diff preview showing exact command strings. Hardcoded confirm primitive renders even if presets are corrupted.

10.3 Audit Trail

Every apply is logged to plugins/CommandPrompterPaper/.editor-audit.jsonl (append-only, JSON Lines):

{"ts":"2026-06-17T12:00:00Z","action":"session.create","initiated_by":"f5a3e2c1-...","token_hash":"sha256:...","mode":"live"}
{"ts":"2026-06-17T12:05:00Z","action":"trust.add","initiated_by":"f5a3e2c1-...","browser_key_hash":"sha256:..."}
{"ts":"2026-06-17T12:10:00Z","action":"apply","initiated_by":"f5a3e2c1-...","confirmed_by":"f5a3e2c1-...","risk_tier":"high","diff":{"added":1,"removed":0,"changed":2},"mode":"live"}

When initiated_by and confirmed_by differ, the log shows a delegation. The token_hash is SHA-256 of the token (not the token itself) for forensic reference without storing the credential.

Network partition during applyIf the change-request frame doesn't reach the plugin (WSS dropped mid-frame), the browser detects the disconnect and shows "Connection lost, retrying." The plugin never received the frame, so no partial apply. If the plugin crashes after receiving the frame but before disk write, the change is lost — the browser can resend on reconnect.
Browser tab closed mid-editWSS disconnects. Plugin detects the disconnect, starts a 60s grace timer. If the browser reconnects within 60s (tab reopen), editing resumes. After 60s, the plugin closes the channel and the session is removed.
Concurrent sessions from same serverAllowed. Each session has its own token and WSS channel. The plugin supports multiple concurrent editor sessions (one per admin). Each session's apply is independent.
High-risk edit with offline initiator and no other editor onlinePending apply held until pending TTL (300s) expires, then dropped. Browser status: "confirmation timed out, no eligible player online." Admin must re-initiate and re-apply when online.
Failed reload after applyPresetRegistry.reload() throws PresetLoadException, previous cache preserved. Plugin sends change-response {state: "rejected", reason: "reload_failed"}. Browser shows the error. The disk file may be in a bad state — the admin should check presets.json manually or use /commandprompter reload after fixing.

The plugin's session persistence (§9.3) reopens the WSS channel. When the plugin rejoins, it pushes the current presets again. The browser's in-progress edits are preserved in memory; the admin can re-apply after the plugin reconnects. If the plugin doesn't reconnect within 60s, the browser shows "Plugin unreachable" and disables Apply.
Relay restart mid-sessionBoth WSS connections drop. The relay has no state — the channel is gone. The plugin's session persistence (§9.3) reopens WSS to /ws/{token}; the channel is recreated. The browser's WSS reconnect logic (automatic retry with backoff) rejoins the same channel. The plugin pushes current presets again. Editing resumes. The trust store is persisted, so no re-trust needed. If the relay is down for >60s, the browser gives up and shows "Relay unreachable."

11.5 Deployment Topology

Plugin: existing Jenkins CI, no change.

Web: GitHub Actions builds a single Docker image containing the relay + SPA static assets. One container, one port, behind Nginx Proxy Manager. The relay serves the SPA at / and the WSS endpoint at /ws/. No separate backend process — the relay is the backend.

Infrastructure: The relay container is the only public-facing service. Nginx exposes it on 443. Proxy-level auth (access list, basic auth, or Cloudflare Access) is recommended in front (§10 mitigation 9). No Redis or database exposure — the relay stores nothing.

12. Out of Scope

  • config.yml / prompt-config.yml editing: This editor is scoped to presets.json only. Runtime settings continue to be edited on disk via /commandprompter reload.
  • Locale .properties files: Managed on disk under plugins/CommandPrompterPaper/locales/ (see 2026-06-15-i18n-system-design.md).
  • Cross-server confirm routing: Confirm is delivered on the server being edited. BungeeCord/Velocity plugin-messaging routing is out of scope. The initiator must join the target server, or another editor confirms.
  • Live Minecraft connection for preview: The preview is a static HTML/CSS mock. It does not connect to a running server or render real player heads.
  • Per-preset permission nodes: Permissions are enforced on the root command that triggers the prompt flow, not in the editor (per preset-configuration-spec.html).
  • Relay-level authentication: The relay is a dumb pipe with no auth. Security is end-to-end (E2E encryption + signatures + trust). Proxy-level auth is a deployment choice, not a relay feature.
  • Multi-admin concurrent editing on the same session: One browser per session (max 2 WSS clients: plugin + browser). Multiple admins edit via separate sessions.
